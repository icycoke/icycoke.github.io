Trie字典树及其应用

Trie概述

  • Trie，又称字典树或前缀树，是一种有序树，用于保存关联数组

  • Trie中的键并不直接保存在节点中，而是取决于该节点在树中的位置

  • 一个节点的所有子孙节点都具有相同的前缀，也就是该节点所对应的值（一般是字符串）

  • 根节点为空节点，从根节点到某一结点所经过的所有节点保存的值相连，为该节点对应的串值

  • 每个节点的所有子节点包含的字符串不相同

  • 如向一个空字典树中插入”panda”，”pain”，”see”则得到如下图所示字典树

  • 优点

    • 在比较时，当前缀不匹配时不继续进行比较，最大的限度地减小了无谓的比较，可用于词频统计和大量字符串排序
    • 和哈希表相比
      • 最坏情况下的时间复杂度比哈希表更好
      • 没有冲突，除非一个键对应多个值
      • 自带排序，使用中序遍历字典树可以得到排序

  • 缺点

    • 以空间换时间，在最坏情况下占大量空间
    • 长浮点数等类型数据会让树的高度变得很大，效率降低

Java并发编程基础复习

创建多线程程序

  • 方法
    1. 继承Thread类
    2. 实现Runnable接口（推荐）
  • 通过实现Runnable接口创建多线程程序的优点：
    • 避免了单继承的局限性
    • 解耦：实现Runnable接口的方式，把设置线程任务和开启新线程进行了分离

线程安全

  • 当多个线程访问了共享数据时，会产生线程安全问题

  • Java引入线程同步机制处理线程安全问题

    1. 同步代码块

    2. 同步方法

      • 非静态方法的锁是实例对象本身（this）
      • 静态方法的锁是本类的class

    3. Lock锁

      • Lock接口提供了比使用synchronized方法和语句可获得的更广泛的锁定操作

线程状态

  • 新建状态
  • 运行状态
  • 阻塞状态
  • 休眠状态
  • 永久等待状态
  • 死亡状态

sleep()和wait()的区别：前者保持对象锁，其他线程无法访问；后者释放对象锁，其他线程可以访问

线程池

  • 如果并发的线程数量很多，并且每个线程完成任务的时间很短，频繁创建线程会大大降低效率，因此需要线程池

  • 流程：

    • 当程序初次启动时，创建多个线程，保存在一个容器中
    • 使用线程时从容器中取线程使用
    • 线程使用完毕后，把线程归还给线程池

  • 线程池的优点

    • 降低资源消耗。每个线程都可被重复利用
    • 提高响应速度。任务到达时，不需要等待线程创建
    • 提高线程的可管理性。可根据系统的承受能力，调整线程池中工作线程的数量

  • JDK1.5之后，内置线程池工厂类Executors，但推荐手动创建线程池。使用该工厂类创建线程池可能会因为允许的请求队列长度或创建线程数量太大而导致内存溢出

  • 通过ThreadPoolExecutor手动创建线程池

    • corePoolSize是线程池维护线程的最少数量，如果任务添加时，线程池中的数量小于该数值，则创建新的线程来处理该任务
    • maximumPoolSize是线程池维护线程的最大数量
    • keepAliveTime是线程池维护线程所允许的空闲时间
    • unit是像城池维护线程所允许的空闲时间的单位
    • workQueue是线程池所使用的缓冲队列
    • handler是线程池对拒绝任务的处理策略

运输层：TCP

可靠数据传输原理

  • 构造可靠数据传输协议

    1. 经完全可靠信道的可靠数据传输：rdt1.0

      • 底层信道是完全可靠的

      • 发送方和接收方的状态机

    2. 经具有比特差错信道的可靠数据传输：rdt2.0，rdt2.1，rdt2.2

      1. rdt2.0

        • 底层信道中的分组可能受损

        • 重传机制的可靠数据传输协议称为自动重传请求（Automatic Repeat reQuest, ARQ）协议

        • ARP协议中需要另外三种协议功能来处理存在比特差错的情况：

          1. 差错检测
          2. 接收方反馈*，包括肯定确认（ACK）与否定确认（NAK）**
          3. 重传**

        • 发送方和接收方的状态机

        • 在发送方处于等待ACK或NAK的状态时，它不能从上层获得更多数据，因此在发送方确信接收方已正确接受当前分组前，发送方将不会发送一块新数据，这种协议被称为停等（stop-and-wait）协议

      2. rdt2.1

        • 为了处理ACK或NAK分组受损的情况，发送方对其数据分组编号，即将发送数据分组的序号放在该字段

        • 接收方需要检查序号确定收到的分组是否是一次重传

        • 发送方和接收方的状态机

      3. rdt2.2

        • 接收方发送上一个正确接收的分组的ACK来代替NAK

    3. 经具有比特差错的丢包信道的可靠数据传输：rdt3.0

      • 底层信道除了比特受损外，还可能丢包

      • 为了实现基于时间的重传机制，设定一个倒计数定时器，超时则重发

      • 由于分组序号在0和1之间交替，因此rdt3.0被称为比特交替协议

      • rdt3.0发送方的状态机

  • 流水线可靠数据传输协议

    • rdt是一个停等协议，因此其性能并不理想
    • 流水线可靠数据传输协议不以停等方式运行，允许发送方发送多个分组而无须等待确认
    • 必须增加序号范围
    • 协议的发送方和接收方需要缓存多个分组
    • 需要解决流水线的差错恢复问题
      • 回退N步（Go-Back-N，GBN）
        • 限制在流水线中未确认的分组数不能超过最大允许数N
        • 因为违背确认的分组的许可序号范围可以被看成是一个在序号范围内长度为N，N也被称为窗口长度，GBN协议也被称为滑动窗口协议
        • 接收方只需要维护下一个按序接受的分组的序号
      • 选择重传
        • 单个分组的差错就会引起GBN重传大量分组，因此需要对重传分组进行选择
        • 接收方将确认一个正确接受的分组而不管其是否按序，失序的分组将被缓存直到所有丢失分组都被收到为止
        • 窗口长度必须小于或等于序号空间大小的一半

TCP（Transmission Control Protocol）概述

  • TCP是面向连接的（connection-oriented），是点对点的

  • TCP提供全双工服务

  • TCP连接建立的三次握手

    1. 客户发送TCP报文，请求建立连接（不承载应用层数据）
    2. 服务器返回TCP报文，同意创建连接（不承载应用层数据）
    3. 客户端返回TCP报文，确认收到同意连接的信号（可承载应用层数据）

    • 作用：

      • 防止服务器端开启无用链接
      • 防止已失效的连接请求报文突然到达

    • 缺点：

      • 必须为每个请求的对象建立和维护一个全新的连接
      • 每个对象经受两倍往返时间（RTT）的交付时延

  • 传输流程：

    1. 客户进程通过socket传递数据流
    2. TCP将数据引导到发送缓存
    3. TCP不时地从发送缓存里取出数据传递到网络层
    4. 数据被传输到接收缓存
    5. 服务器进程通过socket读取接收缓存中的数据
    • TCP可从缓存中取出并放入报文段的数据数量受限于最大报文段长度（Maximum Segment Size, MSS），MSS指在报文段里应用层数据的最大长度，而不是指包括首部的TCP报文段的最大长度
    • MSS通常根据最初确定的由本地发送主机发送到最大链路层帧长度，即最大传输单元（Maximum Transmission Unit, MTU）来设置

TCP报文

  • TCP报文段结构

    源端口号 目的端口号
    序号
    确认号
    首部长度 保留未用 标志字段 接收窗口
    检验和 紧急数据指针
    选项
    数据
    • 首部字段
      • 源端口号和目的端口号。用于多路复用/分解来自或送到上层应用的数据
      • 序号和确认号
        • 序号：该报文的首个字节的字节流编号
        • 确认号：发送方期望收到的下一字节的序号
      • 首部长度
      • 标志字段
        • ACK指示确认字段中的值时有效的
        • RST、SYN和FIN用于连接建立和拆除
        • CWR和ECE用于明确拥塞通告
        • PSH被置位（置为1）时，指示接收方立即将数据交给上层
        • URG用于指示报文段里存在被发送端的上层实体置为“紧急”的数据
      • 接受窗口。用于流量控制，指示接收方愿意接受的字节数量
      • 检验和
      • 紧急数据指针
      • 选项字段。用于发送方和接收方协商最大报文段长度时，或在高速网络环境下用作窗口调节因子时使用
    • 数据字段

TCP的可靠数据传输

  • TCP发送方事件

    事件 发送方动作
    从应用层收到数据 创建序列号是第一个字节编号的Segment
    设置超时时间，开启计时器
    超时 重传引起超时的Segment（单个）
    重启定时器
    收到ACK 如果确认此前未确认的Segment：
         1.更新SendBase
         2.如果窗口中还有违背确认的分组，重新启动定时器

  • TCP接收方事件

    事件 接受方动作
    具有所期望序号的按序报文段到达，所有在期望序号及以前的数据都已经被确认 延迟的ACK，如果下一个按序报文段在500ms内没有到达，则发送一个ACK
    具有所期望序号的按序报文段到达，另一个按序报文段等待ACK传输 立即发送单个累积ACK，以确认两个按序报文段
    比期望序列号大的失序报文段到达，检测出间隔 立即发送冗余ACK，指示下一个期待字节的序号（其为间隔低端的序号
    能部分或完全填充接收数据间隔的报文段到达 倘若该报文段起始于间隔的低端，则立即发送ACK

  • 超时间隔加倍

    • 每次TCP重传时都会将下一次的超时间隔设为先前值的两倍，而定时器在另两个事件（即收到上层应用的数据和收到ACK）中的任意一个启动时，超时间隔由样本RTT和估计RTT推算得到
    • 这一机制防止了网络拥塞情况下，源持续重传分组加剧网络拥塞

  • 快速重传

    • 冗余ACK指再次确认某个报文段的ACK，而发送方先前已经收到对该报文段的确认
    • 一旦收到3个冗余ACK，则立即重传以该ACK字段值为序号的报文段

TCP流量控制

  • TCP为它的应用程序提供了流量控制服务
  • 流量控制是一个速度匹配服务，即发送方的发送速率与接收方的接受速率相匹配
  • 流量控制服务用于消除发送方使接收方缓存溢出的可能性
  • 方法：
    • 发送方维护一个接收窗口，用于给发送方指示接收方还有多少可用的缓存空间
    • 由于TCP是全双工通信，连接两端的发送方都各自维护一个接收窗口
    • 接收方在报文中将接收窗口大小告诉发送方，发送方限制自己已经发送的但还未收到的ACK的数据不超过接收方的接收窗口大小
    • 如果接收方的接收窗口为0时，发送方继续发送只有一个字节数据的报文段，防止被阻塞而不能再发送数据

TCP连接管理

  • TCP连接建立流程（三次握手）
    1. 客户端的TCP首先向服务器端的TCP发送一个特殊的TCP报文段（SYN报文段）。该报文段不含应用层数据，其SYN标志位被置为1。另外，其随机选择一个初始序号，将其置于序号字段中
    2. 服务器端接受SYN报文段，为该TCP连接分配TCP缓存和变量，并向客户端发送允许连接的报文段（SYNACK报文段）。该报文段不含应用层数据，其SYN标志位被置为1，其确认号字段被置为收到的SYN报文段的序号字段值+1。最后，服务器选择自己的初始序号，并将其放置到该报文段的序号字段中
    3. 客户端接受SYNACK报文段，为该TCP连接分配TCP缓存和变量，并向服务器端发送确认的报文段。该报文段可以携带应用层数据，其SYN标志位被置为0，其确认号字段为SYNACK报文段中序号字段值+1
  • TCP连接关闭流程（四次挥手）
    1. 客户端向服务器发送TCP报文段，其中FIN标志位被置为1
    2. 服务器端向客户端发送确认报文段，其中FIN标志位被置为1
    3. 服务器端向客户端发送TCP报文段，其中FIN标志位被置为1
    4. 客户端向服务器段发送确认报文段，其中FIN标志位被置为1

TCP拥塞控制

  • 由于IP不向端系统提供显式的网络拥塞反馈，TCP必须使用端到端拥塞控制，即端系统需要通过对网络行为的观察来推断网络拥塞的存在

  • TCP发送方感知网络拥塞程度并限制其能向连接发送流量的速率

    • TCP发送方跟踪一个额外变量拥塞窗口（congestion window, cwnd），在发送方中违背确认的数据量不会超过拥塞窗口和接收窗口中的最小值
    • TCP的发送方使用报文段成功交付的确认来增加窗口的长度，因此TCP是自计时的

  • TCP拥塞控制原则

    • 一个丢失的报文段意味着拥塞，丢失报文段时应当降低发送方的发送速率
    • 一个确认报文段指示该网络正在向接收方交付发送方的报文段，对先前未确认报文段的确认到达时应当增加发送方的发送速率
    • 带宽探测。TCP调节其传输速率的策略时增加其速率以响应到达的ACK，除非出现丢包事件，此时才减小传输速率

  • TCP拥塞控制算法

    1. 慢启动
      • TCP连接开始时，cwnd的值初始置为一个MSS
      • 每当传输到报文段首次被确认就增加一个MSS
      • 当出现一个由超时指示的丢包事件时，TCP发送方将cwnd设置为1并重新开始慢启动过程，同时将ssthresh（慢启动阈值）置为cwnd/2
      • 当出现一个由3个冗余ACK指示的丢包事件时，ssthresh被更新为cwnd/2，cwnd的值被设置为更新后的ssthresh+3MSS（经过3个RTT接受3个冗余ACK），进入快速恢复状态
      • 当cwnd到达慢启动阈值时，结束慢启动并转移到拥塞避免状态
    2. 拥塞避免
      • 进入拥塞避免模式时，cwnd的值为上次遇到拥塞时的一半，此时距离拥塞可能并不遥远
      • 每经过一个RTT，cwnd的值增加一个MSS
      • 当出现一个由超时指示的丢包事件时，ssthresh被更新为cwnd/2，cwnd的值被设置为1MSS，进入慢启动状态
      • 当出现一个由3个冗余ACK指示的丢包事件时，ssthresh被更新为cwnd/2，cwnd的值被设置为更新后的ssthresh+3MSS（经过3个RTT接受3个冗余ACK），进入快速恢复状态
    3. 快速恢复
      • 每接受到一个冗余ACK，cwnd增加1MSS
      • 当出现一个由超时指示的丢包事件时，ssthresh被更新为cwnd/2，cwnd被设置为1MSS，进入慢启动状态
      • 当发送方接受到一条新的确认报文段时，cwnd被设置为ssthresh，进入拥塞避免状态
    • 为什么在快速恢复状态下接受冗余ACK要将cwnd增加1MSS？ 虽然接收到的是冗余ACK，但仍指示了网络中多了一条报文段的空间，允许发送端再发送一条报文段。由于当前序列号最小的报文丢失，拥塞窗口无法向前移动，因此将cwnd增加1MSS，提高网络利用率

  • 对TCP吞吐量的宏观描述：$$一条连接的平均吞吐量 = \frac{0.75 \times W}{RTT}$$

    • 其中W是丢包事件发生时的窗口长度
    • TCP吞吐量在0.5W/RTT到W/RTT之间线性变化

  • TCP拥塞控制的公平性

    • TCP趋于在竞争的多条TCP连接之间提供对一段瓶颈链路带宽的平等分享

  • 涉及TCP和IP协议的明确拥塞通告

    • 新方案允许网络明确向TCP发送方和接收方发出拥塞信号，在接收方到发送方的ACK报文段中设置ECN回显比特

应用层：DNS

DNS概况

  • DNS是：
    1. 一个由分层的DNS服务器实现的分布式数据库
    2. 一个使得主机能够查询分布式数据库的应用层协议
  • 提供主机名到IP地址的转换
  • DNS协议运行在UDP之上
  • 其它服务：
    • 主机别名
    • 邮件服务器别名
    • 负载分配

DNS数据库

  • DNS服务器是分布式、层次式的
  • DNS服务器分类：
    • 根DNS服务器
    • 顶级域DNS服务器
    • 权威DNS服务器
    • 本地DNS服务器（不属于层次结构中，由ISP提供，起着代理的作用）
  • 查询流程:
    1. 客户与根服务器之一联系，根服务器返回顶级域名服务器的IP地址
    2. 客户与对应顶级域名服务器联系，顶级域名服务器返回权威服务器的IP地址
    3. 客户与对应权威服务器联系，权威服务器返回待查询主机名的IP地址
  • DNS缓存
    • 当某一个DNS服务器接收到一个DNS回答时，它能将映射缓存在本地服务器中
    • 改善了时延性能并减少了因特网上传输到DNS报文数量

DNS记录和报文

  • 资源记录（Resource Record, RR）

    • 格式：（Name, Value, Type, TTL)

      • TTL是该记录的生存时间

      • Type决定了Name和Value值的含义：

        Type Name Value
        A 主机名 该主机名对应的IP地址
        NS 域 知道如何获得该域中主机IP地址的权威DNS服务器的主机名
        CNAME 主机别名 别名为Name的主机的规范主机名
        MX 主机别名 别名为Name的邮件服务器的规范主机名

  • DNS报文

    • 格式

      标识符 标志
      问题数 回答RR数
      权威RR数 附加RR数
      问题（问题的变量数）
      回答（资源记录的变量数）
      权威服务器的记录（资源记录的变量数）
      附加信息（资源记录的变量数）

运输层：UDP

UDP概述

  • UDP（User Data Protocol, 用户数据报协议）是一个非链接的协议
  • UDP使用尽最大努力交付，即不保证可靠交付
  • 使用UDP的应用是可能实现可靠数据传输都，可通过在应用程序自身中建立可靠性机制来实现
  • 主要功能：
    • 多路复用 + 多路分解
    • 少量的差错检查
  • 优点：
    • 关于发送什么数据以及何时发送的应用层控制更为精细（吞吐量不受拥塞控制算法的调节）
    • 无须建立连接
    • 无连接状态（不需要维护连接状态或跟踪参数）
    • 分组首部开销小（仅8字节）

UDP报文

  • 报文结构

    源端口号 目的端口号
    长度（首部+数据） 校验和
    应用数据（报文）

  • 校验和

    • 原理：
      1. 发送方的UDP对报文段中的所有16比特字求和，溢出则回卷
      2. 对该和取反码作为校验和
      3. 接收方将所有的16比特字与校验和求和，若所有位数为1则校验成功
    • 引入原因：
      • 不能保证源和目的之间的所有链路都提供差错检测
      • 遵循端到端原则

应用层：视频流和内容分发网

HTTP流和DASH

  • HTTP流中，视频只是存储在HTTP服务器中作为一个普通的文件
  • 经HTTP的动态适应性流（Dynamic Adaptive Streaming over HTTP，DASH）
    • 在DASH中，视频编码为几个不同的版本，每个版本具有不同的比特率

内容分发网（Content Distribution Network，CDN）

  • CDN管理分布在多个地理位置上的服务器，在他的服务器中存储视频（和其他类型的Web内容，包括文档、图片和音频）的副本，并且试图将每个用户请求定向到一个将提供最好的用户体验的CDN位置。
  • 安置原则：
    1. 深入。通过在遍及全球的接入ISP中部署服务器集群来深入到ISP的接入网中。较高的维护和管理开销。
    2. 邀请做客。通过在少量关键位置建造大集群来邀请ISP做客。较低的维护和管理开销，可能以对端用户的高时延和较低吞吐量作为代价。
  • CDN操作
    • 当用户主机中的一个浏览器指令检索一个特定的视频（由URL标识）时，CDN必须截获该请求，以便能够：
      1. 确定此时适合用于该客户的CDN服务器集群
      2. 将客户的请求重定向到该集群的某台服务器
    • 大多数CDN利用DNS来截获和重定向请求
    • 操作流程：
      1. 用户访问位于某一Web网页
      2. 当用户点击视频链接时，发送一个对对应服务器的DNS请求
      3. 用户的本地DNS服务器将该DNS请求中继到一台用于视频提供服务器的权威DNS服务器，权威DNS服务器返回一个CDN域的主机名
      4. DNS请求进入CDN专用DNS基础设施，用户的本地DNS服务器发送对CDN域内某一主机的DNS请求，CDN的DNS系统返回内容服务器的IP地址
      5. 本地DNS向用户主机转发内容服务CDN节点的IP地址
      6. 客户与目标IP地址的服务器建立TCP链接，并且发出对该视频的HTTP GET请求
  • 集群选择策略
    • 集群选择策略是任何CDN部署的核心
    • 是动态地将客户定向到CDN中某个服务器集群或数据中心的机制
    • 常用策略
      • 地理上最为临近
      • 实时测量

应用层：电子邮件与SMTP

电子邮件系统

  • 组成部分：

    1. 用户代理（user agent）
    2. 邮件服务器（mail server）
    3. 简单邮件传输协议（Simple Mail Transfer Protocol, SMTP）

  • 邮件发送流程：

    1. 发送方撰写邮件，由用户代理传输到发送方的邮件服务器
    2. 发送方的邮件服务器传输到接收方的邮件服务器
    3. 接收方的邮件服务器分发邮件到接收方的邮箱

SMTP概况

  • SMTP是因特网电子邮件中主要的应用层协议
  • SMTP使用TCP可靠数据传输服务
  • SMTP用于从发送方的邮件服务器发送报文到接收方的邮件服务器
  • SMPT要求每个报文采用7比特ASCII码格式，如果某报文包含了非7比特ASCII码字符或二进制数据，则该报文必须按照7比特ASCII码进行编码

邮件报文格式

  • 首部
    • From: 首部行
    • To: 首部行
    • Subject: 首部行
  • 空行分隔
  • 报文体（以ASCII格式表示）

邮件访问协议

  • 用于将邮件从接收方的邮件服务器传送到接收方的用户代理
  • 常见的邮件访问协议
    • 第三版的邮局协议（Post Office Protocol - version 3, POP3）
    • 因特网邮件访问协议（Internet Mail Access Protocol, IMAP）
    • HTTP

应用层：Web和HTTP

HTTP概况

  • Web的应用层协议是超文本传输协议（HyperText Transfer Protocol, HTTP），它是Web的核心。

  • HTTP使用TCP作为它的支撑运输协议

  • HTTP是一个无状态协议，不保存关于客户的任何信息

  • 两种连接方式:

    1. 非持续链接：每个请求/响应对经一个单独的TCP连接发送

    2. 持续链接： 所有的请求及其响应经相同的TCP连接发送

HTTP报文格式

  1. 请求报文
    • 请求行
      • 方法字段
      • URL字段
      • HTTP版本字段
    • 首部行
      • Host
      • Connection
      • User-agent
      • Accept-language
    • 实体体
  2. 响应报文
    • 状态行
      • 版本
      • 状态码
      • 短语
    • 首部行
      • Connection
      • Date
      • Server
      • Last-Modified
      • Cotent-Length
      • Content-Type
    • 实体体

Cookie

  • Cookie技术的四个组件：
    1. 在HTTP响应报文中的一个Cookie首部行（Set-Cookie: xxx）
    2. 在HTTP请求报文中的一个Cookie首部行（Cookie: xxx）
    3. 在用户端系统中保留由一个Cookie文件，由用户的浏览器进行管理
    4. 位于Web站点的一个后端数据库

Web缓存器（代理服务器）

  • 是能够代表初始Web服务器来满足HTTP请求的网络实体
  • 在存储空间中保存最近请求过的对象的副本
  • 条件GET方法
    • 请求报文中使用GET方法且请求报文中包含一个“If-Modified-Since:”
    • 允许缓存器证实它的对象是最新的

计算机网络基本概述

计算机网络结构

  • 网络边缘

    1. 主机（端系统）

      • 位于网络边缘
      • 运行网络应用程序

    2. 客户/服务器（Client/Server）应用模型

      • 客户发送请求，接受服务器响应

    3. 对等（Peer-peer, P2P）应用模型

      • 无（或不仅依赖）专用服务器
      • 通信在对等实体之间直接进行

  • 接入网络

    1. 数字用户线路（DSL）
      • 复用电话线路
      • 接入是独占的
    2. 同轴电缆网络（Cable）
      • 复用电视线路
      • 接入是共享的
    3. 以太网
    4. 无线局域网（LANs）
    5. 广域无线接入

  • 网络核心

    1. 分组交换

      1. 存储转发传输

        • 多数分组交换机在链路的输入端使用存储转发传输：在交换机能够开始向输出链路传输该分组的第一个比特之前，必须接收到整个分组

      2. 排队时延和分组丢失

        • 到达分组在输出缓存中排队等待引起排队时延

        • 缓存空间达到上限，出现分组丢失（丢包）

      3. 转发表和路由选择协议

        • 每台路由器具有一个转发表，用于将目的地址映射成为输出链路

        • 路由选择协议用于自动设置转发表

    2. 电路交换

      1. 复用
        • 时分复用
        • 频分复用
        • 波分复用
        • 码分复用

计算机网络性能

  1. 分组时延
    • 处理时延
    • 排队时延
    • 传输时延
    • 传播时延
  2. 时延带宽积
    • 时延带宽积 = 传播时延 × 带宽
    • 链路的时延带宽积又被称为以比特为单位的链路长度
  3. 吞吐量
    • 表示在发送端与接收端之间传送数据的速率

计算机网络体系

  1. 分层结构

    • 优点：
      • 结构清晰，有利于识别复杂系统的部件及其关系
      • 模块化的分层易于系统更新、维护，任何一层的服务实现的改变对于系统其他层都是透明的，并不影响该系统的其他层
      • 有利于标准化
    • 缺点：
      • 层次过多效率降低，一层可能冗余较低层的功能
      • 某层的功能可能需要仅在其他层才出现的信息，这违反了层次分离的目标

  2. 协议

    • 通信计算机双方必须共同遵从的一组约定，是计算机之间相互通信交流的前提
    • 协议是“水平的”，是控制两个对等实体进行通信的规则的集合
    • 任一层实体需要使用下层服务，遵循本层协议，实现本层功能，向上层提供服务，服务是“垂直的”
  • 下层协议的实现对上层的服务用户是透明的
    • 协议的三要素
      1. 语法（Syntax）
      2. 语义（Semantics）
      3. 时序（Timing）
