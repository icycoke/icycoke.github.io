Java并发编程笔记

Posted on 2020-07-03 Edited on 2020-08-17

1. 线程与进程

  • 进程是系统进行资源分配和调度的基本单位，是程序关于某数据集合的一次运行活动
  • 线程是系统能够运行调度的最小单位
  • 进程是线程的容器，线程是进程中的实际运作单位

2. 多线程程序

2.1 创建多线程程序

  • 方法

    1. 继承Thread类

    2. 实现Runnable接口（推荐）

      Thread类也实现了Runnable接口，因此可以将一个Thread对象作为参数传入另一个Thread对象的构造器中

  • 通过实现Runnable接口创建多线程程序的优点：

    • 避免了单继承的局限性
    • 解耦：将设置线程任务和开启新线程进行了分离

2.2 线程安全

  • 如果某个类被多个线程访问时，这个类总能表现出正确的行为，则这个类是线程安全的

  • 无状态的类一定是线程安全的

  • 竞态条件指输出依赖于不可控的时序，不恰当的执行时序可能导致错误的输出

  • 先检查后执行（Check-Then-Act）是最常见的竞态条件类型

  • Java引入线程同步机制处理线程安全问题

    1. 同步代码块

    2. 同步方法

      • 非静态方法的锁是实例对象本身（this）
      • 静态方法的锁是本类的class

    3. Lock锁

      • Lock接口提供了比使用synchronized方法和语句可获得的更广泛的锁定操作

  • 有些常用方法是存在同步代码块的，例如System.out.println()

2.3 基础方法

2.3.1 start()

  • 开始执行该线程
  • 自动以该线程调用run()方法
  • 如果不使用start()而直接在其他线程中调用该线程的run()方法，则是以其他线程调用

2.3.2 currentThread()

  • 返回代码段正在被哪个线程调用的信息

2.3.3 isAlive()

  • 判断当前线程是否处于活动状态
  • 活动状态指已经启动且尚未终止

2.3.4 sleep()

  • 在指定的毫秒数内让当前正在执行的线程休眠

2.3.5 getId()

  • 取得线程的唯一标识

2.3.6 setName()

  • 设置线程的名称

2.4 停止线程

2.4.1 停止线程的方法

  1. 使用退出标志，使线程正常退出，也就是当run()方法完成后线程终止
  2. 使用stop()方法（不推荐）
    • 线程的中断应由线程自身进行，而不应由其它线程进行
    • 使用stop()会释放锁，可能给数据带来不一致性的结果
  3. 使用interrupt()方法
    • 使用thread.interrupt()将会给thread对象所代表的线程加上中断标志，不会立即中断线程，需要在run()方法中检测中断标志并跳出方法使线程结束
    • 如果令休眠中的线程中断，则会在run()方法中抛出InterruptedException，并且清除中断状态
    • 如果令有中断标志的线程休眠，也会在run()方法中抛出InterruptedException

2.4.1 判断线程是否是停止状态

  • 方法
    1. thread.interrupted()
      • 测试当前线程（执行该语句的线程）是否中断
      • 会清除线程的中断状态
    2. this.isInterrupted()：
      • 测试thread所代表的线程是否中断
      • 不会清除线程的中断状态

2.5 暂停线程

2.5.1 suspend()与resume()方法（不推荐）

  • 极容易造成公共的同步对象被独占，使得其他线程无法访问公共同步对象

2.5.2 yield()方法

  • 令调用该方法的线程放弃当前CPU资源，但放弃的时间不确定

2.6 线程的优先级

  • 线程的优先级分为1~10级，数字越大则优先级越高
  • 使用getPriority()方法获取线程优先级
  • 使用setPriority()方法设置线程优先级
  • 性质
    • 继承性：线程的优先级具有继承性，假设A线程启动B线程，则B线程的优先级与A是一样的
    • 规则性：线程的优先级具有一定规则性，CPU尽量将执行资源让给优先级比较高的线程
    • 随机性：优先级较高的线程不一定总是先执行完

2.7 守护线程

  • Java线程分为用户线程和守护线程
  • 当进程中不存在用户线程时，守护线程自动销毁
  • 只要当前JVM实例中存在任何一个用户线程没有结束，守护线程就仍在工作
  • 通过setDaemon(true)设置该线程为守护线程
  • 例如：GC（垃圾回收器）

3. 对象及变量的并发访问

3.1 synchronized同步方法

  • 同步方法的锁是该方法的实例对象
  • 当A线程获得了某一对象的锁时，B线程必须等待A线程释放锁后才能调用该对象的任意synchronized同步方法，但可以随意调用其他非同步方法

3.1.1 方法内的变量

  • 方法内的变量是私有的，因此方法中的变量不存在非线程安全的问题

3.1.2 实例变量

  • 如果多个线程共同访问1个对象中的实例变量，则有可能出现非线程安全问题
  • 在两个线程访问同一个对象中的同步方法时一定是线程安全的

3.1.3 多个对象多个锁

  • 若是两个线程分别访问同一个类的两个不同实例的同名称同步方法，则JVM创建两个锁，二者相互独立

3.1.4 synchronized锁重入

  • 当一个线程得到一个锁之后，再次请求此锁时可以再次得到该锁，因此在一个同步方法内部调用该对象的其他同步方法时，总能得到锁

3.1.5 出现异常

  • 当一个线程执行的代码出现异常时，该线程持有的锁会自动释放

3.1.6 同步的非继承性

  • 同步本身不具有继承性，父类拥有的同步方法，子类若要继承则需要添加synchronized关键字（重写方法的情况下）

3.2 synchronized同步语句块

  • 同样地，同一个对象锁对应的同步代码块间也存在同步性，当一个线程访问一个有对象锁的同步代码块时，其他线程对同一个对象中的所有其他的同步代码块的访问将被阻塞
  • 同步语句块可以指定锁对象，在有些情况下可以不与其他锁this的同步方法争抢锁，从而提高运行效率

3.2.1 class锁

  • 在使用静态同步方法或synchronized(class)代码块时，给对应的Class类上锁

3.2.2 多线程的死锁

  • 不同的线程都在等待根本不可能被释放的锁时，出现死锁

3.2.3 锁对象的改变

  • 如果同时持有相同的锁对象，则这些线程之间是同步的；如果分别获得锁对象，这些线程之间就是异步的
  • 关键看取得锁时是否为同一个锁对象

3.3 volatile关键字

  • volatile关键字的主要作用时使变量在多个线程间可见
  • volatile保证了数据的可见性，但不保证原子性。在对volatile修饰的变量进行非原子操作（例如自增运算）时，是非线程安全的

3.3.1 volatile与死循环

  • JVM允许每个线程存储变量的副本到私有堆栈中，因此可能会出现异步死循环问题

  • volatile强制从公共堆栈中取得被修饰变量的值，而不是从线程私有数据栈中取得。将上述程序中的isRunning变量用volatile修饰后线程可以正常结束运行

3.3.2 volatile与synchronized

  • volatile使线程同步的轻量级实现，因此性能要优于synchronized
  • volatile只能修饰变量，而synchronized可以修饰方法以及代码块
  • 多线程访问volatile不会发生阻塞，而synchronized会出现阻塞
  • volatile能保证数据的可见性，但不能保证原子性；synchronized能保证原子性，同时在上锁时，清空私有内存中的共享变量并重新从公共内存中获取（与volatile类似的同步的功能），因此间接保证了可见性
  • volatile解决了变量在多个线程中的可见性，而synchronized解决了多个线程之间访问资源的同步性

3.4 原子类

  • 原子操作是指不能被分割的操作，没有其他线程能够中断或检查正在原子操作中的变量，而原子类型就是能被原子操作所用的类型
  • Java的原子类包括AtomicBoolean,AtomicInteger,AtomicLong,AtomicIntegerArray等等，这些类通过CAS原理解决并发情况下的原子性问题

3.4.1 原子类的非线程安全问题

  • 原子类并不是绝对安全的，原子类在具有有逻辑性的情况下输出结果也具有随机性

3.5 CAS（Compare and Swap)

3.5.1 悲观锁

  • 悲观锁总是假设最坏的情况，每次获取数据时都默认其他线程会修改因而上锁
  • 当其他线程想拿到悲观锁上锁数据时，会一直堵塞知道锁被释放
  • synchronized就是悲观锁的一种实现

3.5.2 乐观锁

  • 乐观锁只在更新数据的时候判断一下是否有其它线程修改过该数据
  • CAS就是一种乐观锁的实现方式

3.5.3 CAS原理

  • CAS操作包括内存值V，旧的预期值A，和要修改的新值B这三个操作数
  • 判断V与A的值，当且仅当V与A相等时，将V修改为B，否则什么都不做
  • Java的原子类应用了CAS原理

3.5.4 CAS的优缺点

  • 优点
    • 当同步冲突出现的机会很少时，带来性能提升
  • 缺点
    • ABA问题。CAS仅判断V与A的值相等问题，没有考虑到V经历过A-B-A的更新
    • 循环时间长开销大。如果使用自旋（循环等待）CAS长时间不成功，带来巨大的开销
    • 只能保证一个共享变量的原子操作

3.6 发布与逸出

  • 发布（Publish）是使得对象能够在当前作用域之外的代码中使用。例如，将一个指向该对象的引用保存到其他代码可以访问的地方

  • 当某个不应该发布的对象被发布被称为逸出（Escape），逸出会破坏线程安全性

  • 要注意隐式的this逸出

    • 当对象在构造函数中创建一个线程时，this引用会被新创建的线程共享，因此不应立即启动该线程
    • 当在构造函数中调用一个可改写的实例方法时，会导致this引用在构造过程中逸出

  • 可以使用私有的构造函数搭配公共的工厂方法避免不正确的构造过程

3.7 线程封闭

  • 当访问共享的可变数据时，一种避免使用同步的方式就是不共享数据，尽在单线程内访问数据，这种技术被称为线程封闭

3.7.1 Ad-hoc线程封闭

  • Ad-hoc线程封闭是指，维护线程封闭性的职责完全由程序实现来承担
  • 由于没有任何一种语言特性能将对象封闭到目标线程上，Ad-hoc线程封闭是非常脆弱的
  • 只要能确保只有单个线程对共享volatile变量执行写入操作，即可达成在volatile变量上的特殊的线程封闭

3.7.2 栈封闭

  • 栈封闭是一种特殊的线程封闭，即只能通过局部变量才能访问对象
  • 由于局部变量封闭在执行线程的栈中，其它线程无法访问这个栈

3.7.3 ThreadLocal类

  • ThreadLocal类能使线程中的某个值与保存值的对象关联起来，一个线程只会访问到与它相对应的被关联变量

  • 从概念上看，可以将ThreadLocal<T>视为包含了Map<Thread, T>对象，但在实现上并非如此

  • 为每一个线程都提供了一份变量的副本，从而实现同时访问而互不干扰

  • Thread类由一个ThreadLocalMap类的成员变量，保存了与该线程相关联的变量

  • 核心方法：

    1. set()方法

    2. get()方法

  • ThreadLocalMap的基本结构：

    • 是ThreadLocal的内部类
    • 使用一个Entry[]数组来保存所有的键值对

3.8 不变性

  • 如果某个对象在被创建后其状态就不能被修改，则这个对象就称为不可变对象
  • 不可变对象一定是线程安全的

3.8.1 final关键字

  • final类型的域是不能修改的，同时还能确保初始化过程的安全性，从而可以不受限制地访问不可变对象，并在共享这些对象时无需同步
  • fianl修饰的方法不能被改写
  • final修饰的类不能被继承

4.1 等待/通知机制

4.1.1 wait()和notify()

  • wait()和notify()方法均为Object的实例方法
  • wait()方法令当前线程进行等待，将该其置入预执行队列中，释放当前持有的锁
  • notify()方法通知那些可能等待该对象的对象锁的其他线程中的一个线程，开始等待获取该对象的对象锁，不会立即释放锁，直到退出同步方法/块之后才会释放锁
  • 当线程处于wait()后的状态时，调用线程对象的interrupt()方法会出现InterruptedException异常
  • wait(long)令线程进入等待唤醒状态，若超过long时间则自动唤醒
  • 在调用wait()/notify()方法之前，线程必须获得该对象的对象级别锁，即只能在同步方法/块中调用

3. 线程状态

  • 新建状态
  • 运行状态
  • 阻塞状态
  • 休眠状态
  • 永久等待状态
  • 死亡状态

sleep()和wait()的区别：前者保持对象锁，其他线程无法访问；后者释放对象锁，其他线程可以访问

4. 线程池

  • 如果并发的线程数量很多，并且每个线程完成任务的时间很短，频繁创建线程会大大降低效率，因此需要线程池

  • 流程：

    • 当程序初次启动时，创建多个线程，保存在一个容器中
    • 使用线程时从容器中取线程使用
    • 线程使用完毕后，把线程归还给线程池

  • 线程池的优点

    • 降低资源消耗。每个线程都可被重复利用
    • 提高响应速度。任务到达时，不需要等待线程创建
    • 提高线程的可管理性。可根据系统的承受能力，调整线程池中工作线程的数量

  • JDK1.5之后，内置线程池工厂类Executors，但推荐手动创建线程池。使用该工厂类创建线程池可能会因为允许的请求队列长度或创建线程数量太大而导致内存溢出

  • 通过ThreadPoolExecutor手动创建线程池

    • corePoolSize是线程池维护线程的最少数量，如果任务添加时，线程池中的数量小于该数值，则创建新的线程来处理该任务
    • maximumPoolSize是线程池维护线程的最大数量
    • keepAliveTime是线程池维护线程所允许的空闲时间
    • unit是像城池维护线程所允许的空闲时间的单位
    • workQueue是线程池所使用的缓冲队列
    • handler是线程池对拒绝任务的处理策略
