Computer Security

Posted on 2020-08-27 Edited on 2020-10-11

Intros

What is Computer Security?

  • Computer security is the protection of valued assets of a computer system
  • The value of an asset depends on the owner’s or user’s perspective
  • Also time-dependent

Basic Components of Security

Confidentiality

  • Keeping data and resources hidden
  • Only authorized people or systems can access protected data

Integrity

  • Data integrity (integrity)
  • Origin integrity (authentication)

Availability

  • Enabling access to data and services
  • Achieving availability may require concurrency control and fault tolerance

Threat Classes

Disclosure

  • Unauthorized access to information
    • Snooping: unauthorized interception of information

Deception

  • Acceptance of false data
    • Modification, spoofing, repudiation of origin, denial of receipt

Disruption

  • Interruption or prevention of correct operations

Usurpation

  • Unauthorized control of part of a system
    • Modification, spoofing, delay, denial of service

General Categories of Security Attack

  • Spoofing attack (Disclosure)

    • IP address spoofing
    • ARP spoofing
    • DNS spoofing

  • Snooping/Sniffing (Deception, Disruption, Usurpation)

  • Modification/Alternation (Deception, Usurpation)

  • Repudiation of origin (Deception)

  • Denial of receipt (Usurpation)

  • Delay (Usurpation)

  • Denial of service (Usurpation)

Goals of Security

  • Prevention
    • Prevent attackers from violating security policy
  • Detection
    • Detect attackers’ violation of security policy
  • Recovery
    • Stop attack, assess and repair damage
    • Continue to function correctly even if attack succeeds

Policy and Mechanism

Policy

  • Policy says what is, and is not ,allowed
  • Policy defines “security” for the system/site/etc.
  • If policies conflict, discrepancies may create security vulnerabilities

Mechanism

  • Mechanisms enforces policies
  • Types of mechanisms
    • Secure
    • Precise
    • Broad
  • Trusts on the mechanisms
    • Each mechanism implements part of the security policy
    • The union of security mechanisms implements all aspects of the security policy
    • The mechanisms are implemented correctly
    • The mechanisms are installed and administered correctly

Assurance

  • Provide a basis of “how much” to trust a system
  • Specification
    • Requirements analysis
    • Statement of desired functionality
  • Design
    • How system will meet specification
  • Implementation
    • Programs/systems that carry out design

Operation Issues

  • Cost-benefit analysis
    • Is it cheaper to prevent or recover?
  • Risk analysis
    • Should we protect something?
    • How much should we protect this thing?
  • Laws and customs
    • Are desired security measures illegal?
    • Will people do them?

Access Control

  • An access control mechanism is used to permit a subject to access an object in a particular mode
  • Unauthorized subjects are not permitted to access the object

Terminology

  • Subject: human user or program running on their behalf
  • Object: thing on which an action can be performed
  • Access modes: any controllable actions of subjects on objects

States and State Transition

States

  • States is a collection of the current values of all memory locations, storages, registers, etc.
  • Protection States: a subset of this collection that deals with protection is the protection state of the system. e.g. password used to check access permission

State Transition

  • System changes causes states changes
  • |- represents transition
  • Commands are often called transformation procedures

Access Control Matrix Model

  • Access control matrix is the simplest abstraction mechanism for representing protection state
    • Describes protection state precisely
    • Describes rights of subjects
    • State transitions change elements of matrix
Operations
  • 6 primitive operations alter matrix
    1. create subject
    2. create object
    3. destroy subject
    4. destroy object
    5. enter
      • enter r into A[s, o] means adds r right for subject s over object o
    6. delete
      • delete r from A[s, o] means remove r right from subject s over object o

Safety & Security

  • Safety refers to the abstract model and security refers to the actual implementation
    • A secure system corresponds to a model safe with respect to all rights

Access Control List & Capability List

Access Control List

  • Focus on columns of access control matrix

    • e.g. file1: { (Andy, rx) (Betty, rwxo) (Charlie, rx) }

  • ACLs can be long

    • User ACLs + Abbreviations

  • Revocation:

    • Owner deletes subject’s entries from ACL, or rights from subject’s entry in ACL

Capability List

  • Focus on rows of access control matrix
    • e.g. Andy: { (file1, rx) (file2, r) (file3, rwo) }
  • Revocation:
    • Scanning all C-lists to remove relevant capabilities is far too expensive
    • It’s better to use indirection
      • Each object has entry in a global object table
      • Names in capabilities name the entry, not the object
        • To revoke, zap the entry in the table
        • Can have multiple entries for a single object to allow control of different sets of rights and/or groups of users for each object

ACL v.s. C-List

  • Both theoretically equivalent
    • ACL easily answers given a subject, what objects and how it can access
    • C-List easily answers given an object, what subjects can access it and how
  • ACL-based systems are more common than C-List-based systems

Security Policy Models Intro

What is a Secure System?

  • Starts in authorized state
  • Never enters unauthorized state
    • A breach of security occurs when a system enters an unauthorized state

Access Control Administration

Discretionary Access Control (DAC)

  • Object owners define policy (like in UNIX file system)

Mandatory Access Control (MAC)

  • Environment enforces static policy (like process labeling)

Security Models

  • A security model is a model that represents a particular policy or set of policies
  • Security models focus on specific characteristics of policies
  • There are many different models focus on different characteristics of policies. It’s difficult to perfectly build a model

Confidentiality Policy

  • Goal: prevent the unauthorized disclosure of information
  • Multi-level security (MLS) models are best-known examples
    • Bell-LaPadula Model (BLP model), 1970s

Bell-LaPadula Model (BLP Model)

  • Focus on confidentiality
  • Principle: information cannot flow from high to low

Security Levels

  • Subjects have security clearance
  • Objects have security classification
  • High level subjects can read same level and lower level objects
  • Lower subjects can only write to same level objects or higher level objects (forbid one subject firstly read from high level objects and write to low level objects)

Lattice Model

  • Lattice model expands normal security levels
  • Express by a tuple (clearance Level, Category)

Integrity Policy

BIBA Model

  • No read down, no write up
  • Information flows from top to bottom
  • Focus on who can “write or make change to” a document
  • Integrity labels are different from security labels
    • Security labels limit the flow of information
      • Subjects have security clearance L(s)
      • Objects have security classification L(o)
    • Integrity labels inhibit modification of information
      • Subjects have integrity level I(s)
      • Objects have integrity level I(o)

Clark-Wilson Integrity Model

  • Focus on the consistency and validity of the data

Entities

  • CDI: Constraint Data Item
    • Data subject to integrity controls
  • UDI: Unconstrained Data Item
    • Data not subject to integrity controls
  • IVP: Integrity Verification Procedures
    • Procedures that test the CDIs conform to the integrity constraints
  • TP: Transaction Procedures
    • Procedures that take the system from one valid state to another

Certification Rules

  1. Certification rule 1: When any IVP is run, it must ensure all CDIs are in a valid state
  2. Certification rule 2: For some associated set of CDIs, a TP must transform those CDIs in a valid state into a (possibly different) valid state
  3. …

BIBA Model v.s. Clark-Wilson Model

  • BIBA model
    • No notion of certification rules
    • Trusted subjects ensure actions obey rules
    • Untrusted data examined before being made trusted
  • Clark-Wilson model
    • Explicit requirements that actions must meet
    • Trusted entity must certify method to upgrade untrusted data (and not certify the data itself)

Hybrid Policy

Chinese Wall

  • A model of a security policy that refers equally to confidentiality and integrity
  • Kind of Mandatory Access Control (MAC) model
  • Deals with conflict of interest situations
  • The access of subject can depends on the history of past accesses (dynamic)

Definitions

  • The objects are items of information related to a company
  • A company dataset (CD) contains objects related to a single company
  • A conflict of interest (COI) class contains the datasets of companies in competition
  • CD(O): the company dataset that contains object O
  • COI(O): the COI class that contains object O

Security Conditions

  • Read
    • S can read O if and only if any of the following holds
      • There is an object O’ such that S has accessed O’ and CD(O’) = CD(O)
      • For all objects O’, O’ belongs to PR(S) -> COI(O’) doesn’t equal COI(O)
        • PR(S) is the set of objects S has read
      • O is a sanitized object (i.e., public objects)
      • Initially, PR(s) = ∅, so any initial read request is granted
  • Write
    • Scan write to O if and only if both of the following hold
      • The CW-simple security condition permits S to read O
      • For all objects O’, if Scan read O’, then CD(O’) = CD(O)

Comparison to Bell-LaPadula

  • Fundamentally different
    • CW has no security labels, BLP does
    • CW has notion of “past accesses”, BLP does not
  • BLP can capture state at any time, but cannot track changes over time
  • Access constraints of CW change over time

RBAC: Role-Based Access Control

  • Use role to decide access but not identity
  • Role indicates set of users, is a collection of privileges/permissions associated with some function or affiliation
  • Role hierarchy
    • Partial order of roles
    • Each role gets permissions of roles below
    • List only new permissions given to each role
  • The permissions are static, the user-role membership is transient
  • Not direct MAC and DAC, but may be either of these (i.e.,considering roles as labels)

Cryptography

Cryptosystem

  • Consists of five tuples
    • P: Set of plaintexts
    • K: Set of keys
    • C: Set of ciphertexts
    • E: Set of encryption functions
    • D: Set of decryption functions
  • Attacks
    • Assume adversary knows algorithm used, but not key
    • Three types of attacks
      • ciphertext only: adversary has only ciphertext; goal is to find plaintext, possibly key
      • known plaintext: adversary has ciphertext, corresponding plaintext; goal is to find key
      • chosen plaintext: adversary may supply plaintexts and obtain corresponding ciphertext; goal is to find key (adversary can decide the input and get the output)
    • Basis for attacks
      • Mathematical attacks
        • Based on analysis of underlying mathematics
      • Statistical attacks
        • Make assumptions about the distribution of letters, pairs of letters (digrams), triplets of letters (trigrams), etc.
        • Examine ciphertext, correlate properties with the assumptions

Communication Channel

  • Communication channel is a basic cryptography problem
  • Ingredients for a secure channel
    • Confidentiality: attacker can’t see the message
    • Integrity: attacker can’t modify the message

Classical Cryptography

  • Sender, receiver share common key
    • Keys may be the same, or trivial to derive from one another
    • Sometimes called symmetric cryptography
  • Two basic types
    • Transposition ciphers
    • Substitution ciphers
    • Combinations are called product ciphers

Transposition Cipher

  • Rearrange letters in plaintext to produce ciphertext
  • Usually use anagramming to attack the cipher
  • Terms
    • period: length of key
    • tableau: table used to encipher and decipher
    • polyalphabetic: the key has several different letters
  • The key of a Vigenère cipher should be at least as long as the message and the key must be random

Substitution Cipher

  • Change characters in plaintext to produce ciphertext
  • Usually use exhaustive search or statistical analysis to attack the cipher

DES Algorithm

  • Steps:
    • Divide input into 64-bit blocks
    • Generate 16 round keys
    • Use IP table to generate the Initial Permutation of plaintexts
    • Evenly split IP into $L_0$ and $R_0$
    • Iterate 16 times using round keys and function $f$
